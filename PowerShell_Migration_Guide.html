⚡ PowerShell Laptop Migration Guide

PS 5.1 · PS 7 · VSCode · Corporate OneDrive Builds  |  Last updated: March 2026

🔍 Why This Keeps Happening

Your corporate build redirects Documents to OneDrive. PowerShell uses Documents\WindowsPowerShell\Modules as its user-tier module path. On a new laptop, OneDrive syncs down old/stale modules before system modules are properly set up — causing version conflicts and load failures every time.

⚠️
The classic failure pattern: OneDrive syncs down PowerShellGet 2.2.5 to the user tier, the new laptop has old PowerShellGet 1.0.0.1 at the system tier, and PackageManagement is missing entirely — resulting in CouldNotAutoloadMatchingModule.

Module Path Hierarchy

TierPathUse
User OneDrive\...\WindowsPowerShell\Modules ❌ AVOID OneDrive causes conflicts
System (PS5.1) C:\Program Files\WindowsPowerShell\Modules ✅ ALWAYS USE
System (PS7) C:\Program Files\PowerShell\Modules ✅ ALWAYS USE
Built-in C:\Windows\System32\WindowsPowerShell\v1.0\Modules 🚫 NEVER TOUCH
✅
Golden rule: Always install modules with -Scope AllUsers. This puts them in Program Files and keeps them completely out of OneDrive.

📦 Before You Hand Back the Old Laptop

1

Export Installed Modules List

# Export all system-level modules to CSV
Get-Module -ListAvailable | 
    Where-Object { $_.Path -like "*Program Files*" } |
    Select-Object Name, Version |
    Export-Csv "C:\Temp\InstalledModules.csv" -NoTypeInformation
2

Backup PowerShell Profiles

# See all profile paths
$PROFILE | Select-Object *

# Copy to safe local location (not OneDrive)
Copy-Item $PROFILE.CurrentUserAllHosts "C:\Temp\PS_Profile_Backup.ps1"
3

Export VSCode Extensions

code --list-extensions > "C:\Temp\VSCode_Extensions.txt"
💡
Save these files to a USB stick, email them to yourself, or put them in SharePoint — not OneDrive Documents.

🖥️ New Laptop Setup — Do This First

1

Fix the OneDrive Module Path Permanently

Open PowerShell 5.1 as Administrator and run:

# View current system PSModulePath
[System.Environment]::GetEnvironmentVariable("PSModulePath", "Machine")

# Strip OneDrive from it permanently
$cleanPath = ([System.Environment]::GetEnvironmentVariable("PSModulePath","Machine") -split ';' | 
    Where-Object { $_ -notlike "*OneDrive*" }) -join ';'

[System.Environment]::SetEnvironmentVariable("PSModulePath", $cleanPath, "Machine")
2

Enforce TLS 1.2 (Required on All New Windows Builds)

[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
3

Manually Bootstrap PackageManagement

Since Install-Module won't work yet, download manually in your browser:

🌐
Go to: https://www.powershellgallery.com/api/v2/package/PackageManagement/1.4.8.1 — it will auto-download. Save to C:\Temp\
# Create the target folder
New-Item -ItemType Directory -Force "C:\Program Files\WindowsPowerShell\Modules\PackageManagement\1.4.8.1"

# Rename and extract the .nupkg (it's just a zip)
Copy-Item "C:\Temp\packagemanagement.1.4.8.1.nupkg" "C:\Temp\packagemanagement.zip"
Expand-Archive "C:\Temp\packagemanagement.zip" -DestinationPath "C:\Program Files\WindowsPowerShell\Modules\PackageManagement\1.4.8.1" -Force

# Unblock all extracted files
Get-ChildItem -Recurse "C:\Program Files\WindowsPowerShell\Modules\PackageManagement" | Unblock-File
4

Update PowerShellGet

Import-Module PackageManagement -Force
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
Install-Module PowerShellGet -Scope AllUsers -Force -AllowClobber

Restart PowerShell as Administrator after this step.

5

Reinstall Your Modules from CSV

[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
$modules = Import-Csv "C:\Temp\InstalledModules.csv"
foreach ($m in $modules) {
    Write-Host "Installing $($m.Name) $($m.Version)..." -ForegroundColor Cyan
    Install-Module $m.Name -Scope AllUsers -Force -AllowClobber
}

📝 PowerShell Profile — Add Permanent Protection

Add this to the top of your PS5.1 profile to protect every future session:

# Find your profile path first
$PROFILE

# Then edit it and add these lines at the very top:

# Strip OneDrive from module path to prevent conflicts
$env:PSModulePath = ($env:PSModulePath -split ';' | 
    Where-Object { $_ -notlike "*OneDrive*" }) -join ';'

# Enforce TLS 1.2
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
💡
Store your profile in C:\Program Files\WindowsPowerShell\ or back it up outside OneDrive so it survives laptop changes.

🔵 VSCode — Extra Considerations

How VSCode Differs from a Normal Terminal

BehaviourDetail
Own PS sessionThe PowerShell extension spins up its own process — it reads PSModulePath from Windows environment at launch, not from your terminal
Own profileUses Microsoft.VSCode_profile.ps1 — separate from Microsoft.PowerShell_profile.ps1
PS versionCan silently use a different PS version — check the bottom-left status bar in VSCode
Settings SyncCan restore old extension settings from the cloud, similar to the OneDrive module problem

Harden VSCode's settings.json

Add this to your VSCode settings.json (Ctrl+Shift+P → Open User Settings JSON) to hardcode a clean module path:

{
    "powershell.powerShellAdditionalExePaths": {
        "PS7": "C:\\Program Files\\PowerShell\\7\\pwsh.exe"
    },
    "powershell.powerShellDefaultVersion": "PS7",
    "terminal.integrated.env.windows": {
        "PSModulePath": "C:\\Program Files\\WindowsPowerShell\\Modules;C:\\Program Files\\PowerShell\\Modules;C:\\WINDOWS\\system32\\WindowsPowerShell\\v1.0\\Modules"
    }
}
✅
The terminal.integrated.env.windows entry completely bypasses the OneDrive problem for VSCode's integrated terminal by hardcoding a clean path.

Add the OneDrive Fix to Your VSCode Profile Too

# Find your VSCode-specific profile
$PROFILE.CurrentUserCurrentHost  # run this inside VSCode terminal

# Add the same protection lines at the top of that file
$env:PSModulePath = ($env:PSModulePath -split ';' | 
    Where-Object { $_ -notlike "*OneDrive*" }) -join ';'
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12

Reinstall VSCode Extensions

# Bulk reinstall from your saved list
Get-Content "C:\Temp\VSCode_Extensions.txt" | ForEach-Object {
    code --install-extension $_
}
